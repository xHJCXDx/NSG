## 1. Resumen y Palabras Clave

### Resumen

La gestión de vulnerabilidades (VM) y el parcheo efectivo representan uno de los mayores desafíos en la ciberseguridad moderna, debido al volumen creciente de nuevas *Common Vulnerabilities and Exposures* (CVE) publicadas y la latencia entre su divulgación y la aplicación de la remediación. Esta tesis propone y documenta la arquitectura de un sistema de **inteligencia de vulnerabilidades** automatizado. Utilizando **n8n** como motor de orquestación, se integra la API oficial de la *National Vulnerability Database* (NVD), datos de enriquecimiento de **CVE Details** y capacidades de detección de exposición pública de **Shodan**. Los datos son centralizados y correlacionados con un inventario interno en una base de datos **PostgreSQL**, lo que permite determinar la aplicabilidad real y la priorización basada en el contexto organizacional y el riesgo de explotación. El sistema documenta un flujo de trabajo de cinco etapas y establece un *tracking* de remediación para calcular métricas críticas de seguridad, como el *Mean Time To Detect* (MTTD) y el *Mean Time To Remediate* (MTTR), buscando reducir la ventana de exposición a amenazas críticas. Se presenta el modelo de datos SQL y los anexos técnicos necesarios para su implementación en un entorno real.

### Palabras Clave

Automatización, n8n, Gestión de Vulnerabilidades (VM), *Patch Management*, NVD API, PostgreSQL, CVE, CVSS, SecOps, Shodan.

## 2. Introducción

### 2.1. Contexto

El panorama actual de las ciberamenazas está dominado por la velocidad. Los actores maliciosos aprovechan el breve lapso entre la publicación de una vulnerabilidad crítica y la disponibilidad de un parche. El volumen anual de CVEs supera las 25.000, un número que excede la capacidad de cualquier equipo de seguridad para procesar y priorizar manualmente.

### 2.2. Problema

El problema central radica en la **brecha entre la publicación de la CVE y la remediación efectiva (MTTR)**. La gestión manual no solo es ineficiente, sino que carece de la contextualización necesaria para priorizar: una vulnerabilidad con CVSS alto puede ser irrelevante si no afecta a ningún activo, mientras que una vulnerabilidad media puede ser crítica si el activo está expuesto a Internet o maneja datos sensibles. Se requiere un mecanismo que no solo notifique, sino que filtre, priorice y correlacione automáticamente la información de vulnerabilidades con el inventario de activos de la organización.

### 2.3. Relevancia

La propuesta es relevante porque traslada el proceso de VM de una tarea reactiva, manual y estresante a un flujo de trabajo proactivo, automatizado y basado en el riesgo contextual. El uso de herramientas de código abierto como **n8n** y **PostgreSQL** hace que la solución sea accesible y adaptable a organizaciones con presupuestos limitados o que buscan la soberanía sobre sus herramientas de SecOps.

## 3. Planteamiento del Problema

La dependencia de la revisión humana en la gestión de vulnerabilidades se traduce en altos *Mean Time To Detect* (MTTD) y *Mean Time To Remediate* (MTTR), lo que incrementa significativamente el riesgo de un ciberataque. El desafío es **diseñar e implementar una arquitectura de SecOps automatizada, utilizando n8n, capaz de ingerir, enriquecer, correlacionar y alertar sobre nuevas CVEs en tiempo real, minimizando la intervención manual y proporcionando una métrica objetiva y auditable del progreso de la remediación.**

## 5. Marco Teórico

### 5.1. Fundamentos de Ciberseguridad y Gestión de Riesgos
La tesis se enmarca en la Gestión de Riesgos, enfocándose en la disciplina de **Vulnerability Management (VM)** y **Patch Management (PM)**. Se debe definir el ciclo de vida de una vulnerabilidad, desde su descubrimiento hasta su mitigación.

### 5.2. Estándares de Vulnerabilidades
* **CVE (Common Vulnerabilities and Exposures):** Identificador universal de vulnerabilidades.
* **CVSS (Common Vulnerability Scoring System):** Marco estándar y abierto para cuantificar la severidad de las vulnerabilidades. Se hará hincapié en la versión 3.x/4.x y el uso de métricas Base, Temporal y Ambiental (priorización contextual).
* **CPE (Common Platform Enumeration):** Esquema estandarizado para nombrar sistemas operativos, aplicaciones y dispositivos. Es fundamental para la correlación automática con el inventario.

### 5.3. Fuentes de Inteligencia de Amenazas
* **NVD (National Vulnerability Database):** Repositorio oficial del gobierno de EE. UU. (NIST) que enriquece las CVEs con CVSS, CPE y *vendor statements*. Se debe detallar el uso de su **API 2.0**.
* **CVE Details:** Fuente de datos adicional que proporciona información sobre exploits y métricas de popularidad no siempre disponibles en NVD.
* **Shodan:** Motor de búsqueda para dispositivos conectados a Internet. Se utiliza para enriquecer la vulnerabilidad con el dato de **exposición pública**, es decir, si los activos potencialmente afectados son detectables desde Internet (a través de la **CVEDB API** o búsquedas específicas).

### 5.4. Automatización en SecOps
* **SecOps (Security Operations):** La integración de la seguridad en las operaciones diarias. La automatización es clave para lograr la **velocidad** y **escalabilidad**.
* **n8n como motor de Orquestación:** Plataforma de flujo de trabajo de código abierto (o *low-code*) que permite la integración de APIs, bases de datos y servicios de mensajería sin escribir código desde cero. Su flexibilidad es ideal para el *pipeline* de VM propuesto.

## 9. Desarrollo / Propuesta Técnica

### 9.1. Arquitectura Propuesta

La arquitectura se basa en un modelo centralizado de ingesta, procesamiento y notificación.

$$
\text{NVD API} \xrightarrow[\text{CVE Data}]{\text{n8n - Colector}} \text{PostgreSQL (cves)}
$$

$$
\text{CVE Details/Shodan} \xrightarrow[\text{Enriquecimiento}]{\text{n8n - Enriquecedor}} \text{PostgreSQL (cves)}
$$

$$
\text{PostgreSQL (cves)} + \text{PostgreSQL (software\_inventory)} \xrightarrow[\text{Correlación por CPE}]{\text{n8n - Correlador}} \text{PostgreSQL (applicable\_vulnerabilities)}
$$

$$
\text{applicable\_vulnerabilities} \xrightarrow[\text{Alerta}]{\text{n8n - Alerta}} \text{Slack / Email (SecOps Team)}
$$

### 9.2. Modelo de Datos en PostgreSQL

El modelo de datos es relacional y diseñado para la eficiencia de consulta, utilizando PostgreSQL para la robustez y la capacidad de actualización atómica de registros (`ON CONFLICT DO UPDATE`).

| Tabla | Propósito | Clave Primaria (PK) | Campos Relevantes |
| :--- | :--- | :--- | :--- |
| **`cves`** | Almacena los metadatos de la vulnerabilidad. | `cve_id` (TEXT) | `cvss_score` (NUMERIC), `severity` (TEXT), `description` (TEXT), `cpe_matches` (JSONB), `is_exposed_shodan` (BOOLEAN), `collected_at` (TIMESTAMP) |
| **`software_inventory`** | Catálogo de activos internos. | `asset_id` (SERIAL) | `name` (TEXT), `version` (TEXT), `cpe_23` (TEXT), `ip_address` (TEXT), `owner` (TEXT), `business_criticality` (INT) |
| **`applicable_vulnerabilities`** | Mapea las CVEs a los activos afectados. | `asset_id` + `cve_id` | `detection_date` (TIMESTAMP), `status` (TEXT: 'Pending', 'In Progress', 'Remediated'), `sla_due_date` (DATE), `tracking_owner` (TEXT) |
| **`vulnerability_alerts`** | Histórico de notificaciones generadas. | `alert_id` (SERIAL) | `asset_id` (INT), `cve_id` (TEXT), `alert_level` (TEXT: 'Critical', 'High'), `sent_via` (TEXT), `created_at` (TIMESTAMP) |

### 9.3. Documentación del Workflow de n8n

El flujo principal opera en 5 etapas secuenciales, usando nodos *HTTP Request*, *Function*, *PostgreSQL* y *Slack/Email*.

| Etapa/Nodo | Descripción Técnica (n8n) | Detalle de Operación |
| :--- | :--- | :--- |
| **1. Recolección** | **Nodo `Cron`** -> **Nodo `HTTP Request` (NVD API)** | Llama a `https://services.nvd.nist.gov/rest/json/cves/2.0?lastModifiedStartDate=...`. Utiliza *date ranges* para consultar solo CVEs nuevas o modificadas en las últimas 24 horas. |
| **2. Filtrado/Priorización** | **Nodo `Function`** -> **Nodo `HTTP Request` (Shodan/CVE Details)** | Filtra por CVSS Base Score $\geq$ 7.0. Usa un **Nodo `Function`** para parsear el arreglo `cpeMatch` de la NVD y extraer los `CPE 2.3` aplicables. Luego, consulta la **CVEDB de Shodan** con el `CVE-ID` para enriquecer el registro con `is_exposed_shodan`. |
| **3. Correlación con Inventario** | **Nodo `PostgreSQL` (Select)** -> **Nodo `Function` (Correlation)** | Consulta la tabla `software_inventory` para obtener todos los activos. El **Nodo `Function`** realiza una lógica de *matching* (uso de librerías como `cpe-match` si n8n lo permite, o lógica *regex* simple en JavaScript) para encontrar CPEs coincidentes entre la CVE y el inventario. |
| **4. Alertas Contextualizadas** | **Nodo `PostgreSQL` (Insert)** -> **Nodo `Slack/Email`** | **Inserta/Actualiza** la tabla `applicable_vulnerabilities`. Si el registro es nuevo o cambia de estado, se genera una alerta. La alerta se prioriza: si CVSS $\geq$ 9.0 *O* (CVSS $\geq$ 7.0 **Y** `is_exposed_shodan` = TRUE **Y** `business_criticality` $\geq$ 4), se envía como **CRÍTICA**. |
| **5. Verificación de Parches (*Tracking*)** | **Nodo `Wait`** -> **Nodo `PostgreSQL` (Update)** | Tras la alerta, se usa un **Nodo `Wait`** o un flujo separado. Este verifica periódicamente si el estado del activo en el inventario ha cambiado o si se ha actualizado el campo `status` en `applicable_vulnerabilities` por una acción de remediación externa, actualizando el *tracking* de MTTR. |

### 9.4. Descenso a lo Técnico: Endpoints y Parsing

* **Endpoint NVD API (Ejemplo de Consulta):**
    ```
    GET https://services.nvd.nist.gov/rest/json/cves/2.0?lastModifiedStartDate=2025-11-14T00:00:00.000Z&lastModifiedEndDate=2025-11-15T23:59:59.999Z
    ```
* **Parsing de CPE en n8n (Lógica de Nodo `Function`):**
    El JSON de NVD contiene la información del producto afectado en el campo `configurations[0].nodes[0].cpeMatch`. El nodo `Function` debe iterar sobre este *array* y extraer el valor de `cpe23Uri` para buscar coincidencias.
* **Lógica de ON CONFLICT DO UPDATE en PostgreSQL:**
    El nodo PostgreSQL de la etapa 1 debe utilizar la siguiente sintaxis para evitar duplicados y actualizar información enriquecida (como el score CVSS) si ya existe la CVE:

    ```sql
    INSERT INTO cves (cve_id, cvss_score, severity, description, collected_at, cpe_matches)
    VALUES (
        '{{ $json.cveId }}',
        '{{ $json.cvss_score }}',
        '{{ $json.severity }}',
        '{{ $json.description }}',
        NOW(),
        '{{ JSON.stringify($json.cpe_matches) }}'
    )
    ON CONFLICT (cve_id) DO UPDATE
    SET
        cvss_score = EXCLUDED.cvss_score,
        severity = EXCLUDED.severity,
        collected_at = NOW();
    ```

## 13. Limitaciones y Trabajo Futuro

### 13.1. Limitaciones

1.  **Precisión del Inventario (CPE):** La precisión del sistema depende directamente de la calidad y el formato estandarizado (CPE 2.3) del inventario (`software_inventory`). Un inventario desactualizado o mal formateado generará una correlación incorrecta.
2.  **API Rate Limits:** La dependencia de APIs externas (NVD, Shodan) está sujeta a sus límites de peticiones, lo que puede requerir una arquitectura de colas de mensajes o estrategias de espera más complejas dentro de n8n.

### 13.2. Trabajo Futuro

1.  **Automatización del Parcheo:** Extender el *workflow* de n8n para interactuar con herramientas de *Patch Management* (*RMM/MDM*) como Ansible o SCCM, para iniciar automáticamente la tarea de despliegue del parche.
2.  **Análisis de Impacto (Dependencias):** Integración con gráficos de dependencias de *software* para evaluar el riesgo en cascada y evitar la falla de servicios críticos al aplicar un parche.

## 15. Anexos

### Anexo A: Script SQL (PostgreSQL)

```sql
-- Creación de la tabla de metadatos de CVEs
CREATE TABLE cves (
    cve_id TEXT PRIMARY KEY,
    cvss_score NUMERIC(3, 1),
    severity TEXT, -- Mapeado de CVSS Score (Critical, High, Medium, Low)
    description TEXT,
    cpe_matches JSONB, -- Almacena la estructura de CPE de NVD para correlación
    is_exposed_shodan BOOLEAN DEFAULT FALSE, -- Flag de exposición pública
    collected_at TIMESTAMP WITHOUT TIME ZONE
);

-- Índices para optimizar búsquedas y filtrados
CREATE INDEX idx_cves_score_severity ON cves(cvss_score DESC, severity);
CREATE INDEX idx_cves_collected_at ON cves(collected_at DESC);

-- Creación de la tabla de inventario de software (ejemplo)
CREATE TABLE software_inventory (
    asset_id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    version TEXT NOT NULL,
    cpe_23 TEXT UNIQUE, -- CPE estandarizado para correlación
    ip_address INET,
    owner TEXT,
    business_criticality INT NOT NULL, -- 1 (Low) a 5 (Critical)
    updated_at TIMESTAMP WITHOUT TIME ZONE
);

-- Creación de la tabla de vulnerabilidades aplicables (el resultado de la correlación)
CREATE TABLE applicable_vulnerabilities (
    asset_id INT REFERENCES software_inventory(asset_id),
    cve_id TEXT REFERENCES cves(cve_id),
    detection_date TIMESTAMP WITHOUT TIME ZONE NOT NULL,
    status TEXT NOT NULL DEFAULT 'Pending', -- Pending, In Progress, Remediated
    sla_due_date DATE,
    tracking_owner TEXT,
    remediation_notes TEXT,
    PRIMARY KEY (asset_id, cve_id) -- Clave compuesta
);

-- Creación de la tabla de histórico de alertas
CREATE TABLE vulnerability_alerts (
    alert_id SERIAL PRIMARY KEY,
    asset_id INT REFERENCES software_inventory(asset_id),
    cve_id TEXT REFERENCES cves(cve_id),
    alert_level TEXT NOT NULL, -- Critical, High, Medium
    sent_via TEXT, -- Slack, Email
    alert_message TEXT,
    created_at TIMESTAMP WITHOUT TIME ZONE NOT NULL
);

-- Índices de tracking para cálculo de MTTR/MTTD
CREATE INDEX idx_applicable_status_date ON applicable_vulnerabilities(status, detection_date);
```

### Anexo B: Estructura del Workflow n8n

| Flujo Principal: `VM_Intelligence_Orchestrator` |
| :--- |
| 1. **Trigger:** `Cron` Node (ej. Cada 30 minutos) |
| 2. **Recolección NVD:** `HTTP Request` (GET NVD API 2.0 con `lastModifiedStartDate`) |
| 3. **Parseo y Normalización:** `Function` Node (Extrae CVE ID, Score, y mapea `cpe23Uri` y `is_exposed_shodan` a JSON unificado) |
| 4. **Enriquecimiento Shodan:** `HTTP Request` (GET CVEDB API por cada CVE ID, si el score es $\geq 7.0$) |
| 5. **Actualización DB:** `PostgreSQL` Node (Usa `INSERT...ON CONFLICT DO UPDATE` en la tabla `cves`) |
| 6. **Correlación:** `PostgreSQL` Node (Consulta cruzada de `cves` y `software_inventory` por patrón `cpe_23`) |
| 7. **Lógica de Priorización/SLA:** `Function` Node (Calcula `sla_due_date` y `alert_level` basado en CVSS y `business_criticality`) |
| 8. **Actualización Aplicabilidad:** `PostgreSQL` Node (`INSERT/UPDATE` en `applicable_vulnerabilities`) |
| 9. **Notificación Condicional:** `IF` Node (Condición: `alert_level` es 'Critical' o 'High' **Y** `status` es 'Pending') |
| 10. **Alerta Slack:** `Slack` Node (Solo si la condición se cumple) |
| 11. **Registro Alerta:** `PostgreSQL` Node (Inserta registro en `vulnerability_alerts`) |
